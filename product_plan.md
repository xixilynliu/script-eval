# 剧本诊断产品化方案

## 一、产品定位

### 一句话定位
**面向编剧/创作者的 AI 剧本诊断工具**——上传剧本，获得专业级商业潜力诊断报告和具体改稿建议。

### 核心价值主张
| 用户痛点 | 产品价值 |
|---------|---------|
| 写完剧本不知道好不好，只能靠朋友帮看 | AI 秒出专业评估，17个维度全面诊断 |
| 投稿被拒不知道原因，改稿没方向 | 每个维度给出具体可打磨点，直接指导改稿 |
| 请专业编审看稿要钱、要等、还要欠人情 | 随时可用、价格低、无社交压力 |
| 不了解市场趋势，写的东西不符合市场需求 | 自动对标爆款模式，判断市场适配度 |

### 与竞品的差异
| 维度 | 创一AI·剧本医生（参考PDF） | 我们的产品 |
|------|--------------------------|----------|
| 目标用户 | 制作公司（筛稿决策） | 编剧/创作者（改稿提升） |
| 核心功能 | 打分 + 判断值不值得投 | **诊断 + 教你怎么改** |
| 报告调性 | 投资评估报告（冷静客观） | 改稿诊断书（建设性、鼓励性） |
| 使用频率 | 单次筛稿 | 反复迭代（改一次、诊断一次） |

## 二、核心用户旅程

```
首次使用：
  注册 → 上传剧本（粘贴/上传文件）→ 等待诊断 → 阅读报告 → 根据建议改稿

反复使用（核心留存动作）：
  改稿 → 再次上传 → 对比两次诊断 → 看到分数提升 → 继续改
                                    ↑
                            这是留存的关键钩子
```

### 用户使用场景

**场景1：投稿前自检**
> 编剧小王写完一个甜宠短剧，准备投给制作公司。投稿前用诊断工具跑一遍，发现"钩子强度"只有 62 分，报告指出"第一集前30秒缺乏冲突设置"。小王按建议改了开头，再跑一次，钩子强度提升到 78 分。

**场景2：改稿参考**
> 编剧小李的古装权谋剧被制作公司退稿，说"节奏太慢"。她不知道具体怎么改。上传到诊断工具，发现"节奏与步调"68分，报告明确指出第3、5、8集的哪些对话场景过长，建议压缩到多少。

**场景3：学习提升**
> 新人编剧小张想了解自己的写作短板。连续诊断了5个剧本后，发现自己的"对白质量"和"角色吸引力"一直偏低，而"情节连贯性"是强项。他知道了自己该重点提升的方向。

## 三、产品功能设计

### MVP（最小可行产品）

只做核心闭环，砍掉一切非必要功能：

```
┌──────────────────────────────────────┐
│  上传/粘贴剧本  →  AI 诊断  →  查看报告  │
└──────────────────────────────────────┘
```

**MVP 功能清单：**

| 功能 | 优先级 | 说明 |
|------|--------|------|
| 剧本上传 | P0 | 支持粘贴文本 + 上传 .txt/.docx 文件 |
| AI 诊断 | P0 | 调用评估引擎，产出结构化结果 |
| 诊断报告页 | P0 | 总分 + 四大板块 + 17维度评分 + 可打磨点 + 建议 |
| 用户注册/登录 | P0 | 手机号/微信登录 |
| 历史报告 | P1 | 查看过去的诊断记录 |
| 免费额度 | P0 | 新用户送1-3次免费诊断 |

**MVP 不做的：**
- 不做收稿匹配（这是制作公司功能）
- 不做多版本对比（V2 做）
- 不做社区/分享（V2 做）
- 不做改稿 AI（V3 做）

### V2（验证PMF后）

| 功能 | 价值 |
|------|------|
| 多版本对比 | 上传改稿后，与上一版对比，显示哪些维度提升/下降 |
| 逐集诊断 | 除了整体报告，支持查看单集的详细诊断 |
| 题材选择 | 用户可指定题材类型（甜宠/虐恋/悬疑...），获得更精准的评估 |
| 报告导出 | 导出 PDF 报告（可作为投稿附件） |
| 诊断历史趋势 | 可视化显示多次诊断的分数变化趋势 |

### V3（规模化后）

| 功能 | 价值 |
|------|------|
| AI 改稿建议 | 不仅告诉你问题在哪，还给出具体的改写示例 |
| 爆款剧本库 | 收录市场爆款剧本的评估数据，作为参考基线 |
| 创作者社区 | 编剧交流、互评、接单 |
| 制作公司入口 | B端功能：征稿+批量筛稿（开辟B端收入） |

## 五、技术架构

### 整体架构

```
┌─────────┐     ┌──────────┐     ┌──────────────┐     ┌─────────┐
│  Web前端  │────▶│  后端API   │────▶│  评估引擎      │────▶│  LLM API │
│ (React)  │◀────│ (FastAPI) │◀────│ (script_eval) │◀────│ (文心等)  │
└─────────┘     └──────────┘     └──────────────┘     └─────────┘
                      │
                      ▼
                ┌──────────┐
                │  数据库    │
                │ (MySQL/   │
                │  Postgres)│
                └──────────┘
```

### 后端核心模块

```
backend/
├── app/
│   ├── main.py              # FastAPI 入口
│   ├── api/
│   │   ├── auth.py          # 登录注册
│   │   ├── scripts.py       # 剧本上传
│   │   ├── evaluations.py   # 诊断任务管理
│   │   └── reports.py       # 报告查询
│   ├── services/

│   │   ├── file_parser.py   # 文件解析（txt/docx/pdf → 纯文本）
│   │   └── report_service.py # 报告生成与存储
│   ├── models/
│   │   ├── user.py          # 用户模型

│   └── core/

│       └── security.py      # 认证
```

### 关键 API 设计

```
POST   /api/auth/login           # 登录
POST   /api/scripts/upload       # 上传剧本 → 返回 script_id
POST   /api/evaluations/create   # 创建诊断任务 → 返回 eval_id（异步）
GET    /api/evaluations/{id}     # 查询诊断状态和结果
GET    /api/evaluations/{id}/report  # 获取完整报告
GET    /api/users/me/evaluations # 历史诊断列表
```

### 异步处理（关键）

诊断过程需要 6 次 LLM 调用，耗时 30-90 秒。必须做异步：

```
用户点击"开始诊断"
      │
      ▼
后端创建诊断任务（状态: pending）→ 立即返回 eval_id
      │
      ▼
后台 Worker 执行诊断（Celery / asyncio）
      │
      ├── 进度: 20% → 基础信息提取完成
      ├── 进度: 40% → 叙事评估完成
      ├── 进度: 60% → 市场评估完成
      ├── 进度: 80% → 合规+商业评估完成
      └── 进度: 100% → 建议生成完成
      │
      ▼
前端轮询 GET /evaluations/{id} 获取进度
诊断完成后展示报告
```

### 数据库模型

```sql
-- 用户表
users (id, phone, nickname, free_quota, created_at)

-- 剧本表
scripts (id, user_id, title, content, file_url, char_count, created_at)

-- 诊断结果表
evaluations (
  id, user_id, script_id,
  status,          -- pending / processing / completed / failed
  progress,        -- 0-100
  total_score,
  total_grade,
  result_json,     -- 完整 EvalResult 的 JSON
  report_md,       -- 生成的 Markdown 报告
  created_at, completed_at
)
```

## 七、落地路径

### Phase 1：验证需求（1-2周）

**目标**：验证编剧是否愿意为 AI 诊断付费

**做法**：
- 不做 Web 前端
- 用现有的 `script_eval.py` 手动帮 10-20 个编剧诊断
- 把报告发给他们，收集反馈：
  - 报告有没有说到点上？
  - 哪些维度最有价值？哪些无感？
  - 愿意为此付多少钱？
  - 会不会改完再来诊断？

### Phase 2：MVP 上线（2-4周）

**目标**：最小闭环产品上线

**技术栈**：
- 前端：React + Tailwind（或直接用低代码平台快速搭）
- 后端：FastAPI + Celery
- 数据库：PostgreSQL
- 部署：单台云服务器即可

**功能**：上传 → 诊断 → 报告 → 登录 → 历史记录

### Phase 3：优化迭代（持续）

基于用户数据和反馈：
- 优化评估准确度（调 prompt、调权重）
- 增加多版本对比
- 增加逐集诊断
- 增加 PDF 导出
- 优化报告的"改稿指导性"

## 八、关键风险和应对

| 风险 | 影响 | 应对 |
|------|------|------|
| AI 评估不准，用户不信任 | 用户不付费、不复购 | Phase 1 充分验证，持续调优 prompt；报告中标注"AI辅助参考" |
| LLM 输出不稳定（JSON解析失败等） | 用户体验差 | 已有容错解析；增加重试机制；做好降级方案 |
| 剧本版权/隐私担忧 | 用户不敢上传 | 明确隐私政策：不存储原文/不用于训练；支持诊断后删除 |
| 单次成本高，免费用户亏损 | 商业模式不成立 | 严格控制免费额度；优化 prompt 减少 token 消耗 |
| 创一AI等竞品先发 | 市场被占 | 差异化定位：我们服务创作者改稿，不是服务制作公司筛稿 |
